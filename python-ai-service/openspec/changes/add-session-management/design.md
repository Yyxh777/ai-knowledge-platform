## Context

当前 Python AI 服务具备基础聊天与工作流能力，但会话状态与历史消息管理未形成统一领域模型。现状导致前端难以稳定展示会话列表、继续既有会话、以及按用户维度隔离历史数据。约束条件是尽量复用现有 FastAPI、SQLite 与认证中间件能力，避免引入新基础设施，并保持与现有聊天接口兼容。

## Goals / Non-Goals

**Goals:**
- 定义清晰的会话与消息数据模型，支持会话生命周期管理与历史查询。
- 为聊天接口提供 `session_id` 兼容扩展，支持“显式会话”与“自动建会话”两种模式。
- 确保会话与消息访问受用户身份约束，避免跨用户数据泄露。
- 为后续实现会话检索、归档和统计分析预留可扩展字段与索引。

**Non-Goals:**
- 不实现跨设备实时同步或 WebSocket 会话推送。
- 不在本次引入向量化会话检索、摘要生成或长期记忆策略。
- 不替换当前主要存储引擎（仍使用现有 SQLite 方案）。

## Decisions

1. 会话与消息采用双表建模（`chat_sessions` + `chat_messages`）。
   - Rationale: 会话元数据与消息明细解耦，便于分页、统计和清理策略。
   - Alternatives:
     - 单表 JSON 存储：实现快但查询与索引能力弱。
     - 外部数据库（MySQL/PostgreSQL）：扩展性更好但超出当前范围。

2. 所有读写接口以 `user_id + session_id` 作为权限校验主键。
   - Rationale: 在应用层实现稳定隔离，避免仅凭 `session_id` 访问他人历史。
   - Alternatives:
     - 仅靠中间件注入用户上下文，不做服务层二次校验：风险较高。

3. 聊天接口保持向后兼容：请求未携带 `session_id` 时自动创建会话并返回。
   - Rationale: 老客户端无需改造即可继续调用，新客户端可逐步接入会话管理。
   - Alternatives:
     - 强制要求传入 `session_id`：实施简单但会造成兼容性断裂。

4. 历史消息查询按 `created_at` 升序返回，并提供分页参数。
   - Rationale: 满足对话回放顺序与性能控制需求，避免一次性加载大历史。
   - Alternatives:
     - 全量返回：实现简单但会导致长会话响应膨胀。

## Risks / Trade-offs

- [SQLite 并发写入瓶颈] -> 通过短事务、必要索引和分页读取降低锁竞争；后续保留迁移到外部 DB 的接口边界。
- [自动创建会话可能导致会话碎片化] -> 通过前端缓存最近 `session_id` 与后端会话命名/更新时间策略改善聚合。
- [历史数据增长导致查询变慢] -> 增加 `(user_id, updated_at)`、`(session_id, created_at)` 索引，并在任务中加入归档/清理预留点。
- [权限校验遗漏] -> 将校验集中在 service 层公共方法，避免各 API 路由重复实现造成漏检。

## Migration Plan

1. 新增会话与消息表（含索引），提供幂等初始化脚本。
2. 增加会话管理 API 与消息查询 API，不影响现有聊天路径。
3. 扩展聊天接口输入/输出以支持 `session_id`，默认自动创建。
4. 灰度联调前端会话列表与历史回放；观察数据库体量和响应时间。
5. 如异常，回滚为“仅保留现有聊天流程”并停用新会话入口，数据库新增表可保留不影响旧逻辑。
