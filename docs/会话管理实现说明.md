# AI 知识平台 · 会话管理实现说明

本文档汇总本项目「会话管理」相关能力的**目标架构、实现位置、数据流、配置与演进**，便于复盘与后续迭代（与 Agent / 人工协作时可作为上下文附件）。

---

## 1. 设计目标

| 维度 | 说明 |
|------|------|
| **会话标识** | `thread_id` 格式 `{userId}_{毫秒时间戳}`，同一会话生命周期内**不变**；新建会话时生成一次，历史续聊沿用同一 ID。 |
| **图内记忆** | LangGraph 运行态以 **PostgreSQL checkpoint** 为准（按 `thread_id` 存取），多轮上下文延续依赖 checkpoint，**不从 MySQL 回填 messages 到初始 state**（避免与 checkpoint 重复合并）。 |
| **业务展示** | **MySQL** 存储 `conversations` + `messages`，供列表、详情、审计；由 **Python 写入**，前端列表曾规划由 Java 提供，当前 **历史列表/消息 REST 在 Python** 实现，静态页直连 Python。 |
| **鉴权** | WebSocket 与 HTTP 历史接口均通过 **Blade token** 调 Java 校验；`thread_id` 须满足前缀 `{user_id}_` 且库内归属一致（若已有会话行）。 |

## 3. 存储分工

### 3.1 MySQL（业务会话与消息落库）

- **表**：`conversations`（id = `thread_id`）、`messages`（conversation_id、role、content、时间戳等）。
- **写入**：Python 在流式对话中 **先写用户消息 → 执行图 → 再写助手消息**（见 `rag_service.chat_with_rag_stream`）。
- **读取**：`GET /ai/conversations`、`GET /ai/conversations/{thread_id}/messages`（仅所有者）。

建表 SQL 口径与项目内 DDL 保持一致（字段名、Unix 秒级时间戳等）。

### 3.2 PostgreSQL（LangGraph Checkpoint）

- 使用 **`langgraph-checkpoint-postgres`** 的 `AsyncPostgresSaver` + **`psycopg` 连接池**。
- 应用启动时 **`await checkpointer.setup()`** 创建/迁移 checkpoint 表（与 MySQL 业务表分离）。
- 环境变量 **`POSTGRES_URI`**（如 `postgresql://user:pass@host:5432/dbname`）。

### 3.3 与旧实现的差异（备忘）

- **曾用**：SQLite 文件库 + Redis checkpoint；**现为**：MySQL + PostgreSQL。
- **曾用**：每次请求临时 `thread_id__ckpt_时间戳` 避免与 SQLite 历史重复；**现为** 稳定 `thread_id`，**不再** 向 `initial_state["messages"]` 注入 MySQL 历史。

## 4. Python 服务实现要点

### 4.1 配置 [`python-ai-service/config.py`](python-ai-service/config.py)

- `POSTGRES_URI`：checkpoint。
- `MYSQL_*`：会话落库与历史查询。
- Redis 相关配置若保留，仅作其它用途标注，**不作为** LangGraph 记忆。

### 4.2 生命周期与图编译 [`python-ai-service/service/rag_service.py`](python-ai-service/service/rag_service.py)

- `startup()`：`AsyncConnectionPool` → `AsyncPostgresSaver` → `setup()` → `create_classification_graph(checkpointer=...)`。
- `shutdown()`：关闭连接池。
- `chat_with_rag_stream`：
  - `config["configurable"]["thread_id"]` = 业务 `thread_id`（无二阶 ckpt id）。
  - `initial_state["messages"] = []`。
  - 顺序：`ensure_conversation` → `add_message(user)` → `astream_events` → 成功后 `add_message(assistant)`。

### 4.3 数据访问 [`python-ai-service/utils/conversation_db.py`](python-ai-service/utils/conversation_db.py)

- `pymysql` 短连接；`ensure_conversation`、`add_message`、`list_conversations_for_user`、`list_messages_for_user`、`validate_thread_access`（WebSocket 用）。
- `init_db()` 为空（表由运维/DDL 维护）。
- `load_messages` 保留供极端场景（如 checkpoint 恢复策略），**聊天主路径不调用**。

### 4.4 WebSocket [`python-ai-service/api/chat.py`](python-ai-service/api/chat.py)

- 首包校验 token；解析 `thread_id`（缺省则生成 `{user_id}_{ms}`）。
- `validate_thread_access(thread_id, user_id)` 通过后再进入消息循环。

### 4.5 历史 REST [`python-ai-service/api/conversation.py`](python-ai-service/api/conversation.py)

- `GET /ai/conversations?limit=`：当前用户会话列表。
- `GET /ai/conversations/{thread_id}/messages`：会话消息（无权重返回 404）。
- Header：`Authorization: Bearer ...` 或 `Blade-Auth`；内部 `validate_token`。

### 4.6 图内生成与流式 [`python-ai-service/workflows/rag_graph.py`](python-ai-service/workflows/rag_graph.py)

- **会话记忆**：节点仍向 `state["messages"]` 追加本轮 Human/AI；reducer `_trim_messages` 限制轮数。
- **打字机效果**：生成类节点须 **`streaming=True`** 且使用 **`llm.stream(..., config=config)`** 拼接（勿对生成路径使用 `invoke`），否则 `astream_events` 难以收到 `on_chat_model_stream`，前端会整段返回。
- 涉及节点：`generate_answer_with_rag`、`generate_answer_with_db`、`direct_answer`（签名含 `RunnableConfig`）。

### 4.7 依赖 [`python-ai-service/requirements.txt`](python-ai-service/requirements.txt)

- `langgraph-checkpoint-postgres`、`psycopg[binary]`；已移除对 LangGraph 的 `langgraph-checkpoint-redis` 依赖（若未做其它用途）。

## 5. 前端（静态页）要点

**文件**：[`java-service/src/main/resources/static/chat.html`](java-service/src/main/resources/static/chat.html)

| 能力 | 说明 |
|------|------|
| WebSocket | `ws://<python-host>:8000/ai/ws/chat`，首包带 `token`，续聊带 `thread_id`。 |
| 历史 API | `http://localhost:8000/ai/conversations`（可按部署改常量 `PYTHON_API`）。 |
| 侧栏 | 历史列表、点击加载消息、`connectWebSocket(threadId)` 续聊；**新对话**清空并重连不带 `thread_id`。 |
| 侧栏显隐 | 顶栏按钮切换 `history-collapsed`，`localStorage` 键 `chat_history_sidebar_visible`。 |
| 已知交互坑 | 发送前若不存在 `#emptyState`，勿裸调 `emptyState.style`（已做判空）。 |

**静态 URL**：Spring Boot 默认 `classpath:/static/` 映射到站点根路径，推荐使用 **`/chat.html`**；已为 **`/static/chat.html`** 增加资源映射（见 `BladeConfiguration.addResourceHandlers`）。

## 6. Java 服务边界

- **会话列表未在 Java 实现**（需求调整为 Python 提供 REST）。
- **`ChatController`** 保持原有非会话扩展接口（如 `POST /chat`），与会话管理解耦。
- **OAuth**：Python 通过已有 Java 用户信息接口校验 token，前后端与 Blade 一致。

---

## 7. 端到端流程（续聊）

1. 用户登录 Blade，静态页带 `access_token`。
2. 打开聊天页：连接 WS，首包 `{ token }` → 服务端返回 `auth_success` + `thread_id`。
3. 用户提问：Python 写 MySQL 用户行 → LangGraph 用 **PG checkpoint** 续写 → 流式返回 → 写 MySQL 助手行。
4. 用户点开侧栏历史：`GET .../messages` 渲染 → `disconnect` → `connectWebSocket(saved_thread_id)` → 后续提问同一条 `thread_id`。
