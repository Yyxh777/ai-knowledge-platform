## 全局问题扫描结果

我在你的代码里找到以下问题，按严重程度分级：

### 🔴 严重 Bug（会造成数据损坏或功能失效）

**1. chunk ID 碰撞（`document_service.py` 第122行）**
```python
# 当前: 每个文档的chunk id都从0开始
data.append({"id": i, ...})
```
上传第二个文档时，`id=0,1,2...` 会直接覆盖第一个文档的数据，Milvus主键冲突，旧数据消失。这是数据层面的严重bug。

**2. user_info 作用域问题（`chat.py` 第99行）**
```python
except WebSocketDisconnect:
    print(f"用户 {user_info.get('account', '未知')} 断开连接")
```
如果用户在 auth 握手阶段就断开连接，`user_info` 根本没有被赋值，这里会直接 `NameError`。

**3. role_id KeyError（`rag_service.py` 第71行）**
```python
"role_id": user_info["role_id"],   # 用[]不用.get()
"role_name": user_info["role_name"],
```
Java 返回的 user_info 里如果 roleId/roleName 为空，直接 KeyError 崩掉。

---

### 🟡 设计问题（功能能用，但不对）

**4. 伪流式输出（`rag_service.py` 第86-88行）**
```python
# LLM已经生成完整答案，再手动切成5字一段
for i in range(0, len(answer), chunk_size):
    yield answer[i:i + chunk_size]
```
这不是流式输出，是批量输出模拟流式。用户等待时间 = 全文生成时间，体验和非流式完全一样，毫无意义。

**5. Milvus 无显式 Schema（`milvus_utils.py` 第13-21行）**
```python
milvus_client.create_collection(
    collection_name=collection_name,
    dimension=1024,
    # 没有定义 type、access_level、record_id 字段类型
)
```
用的是 Milvus auto schema，`type`、`access_level` 等字段没有标量索引，`filter='type == "policy"'` 这类查询做的是全量扫描，性能极差。数据量大之后检索会越来越慢。

**6. 每个节点都重新创建 LLM 和 Milvus 客户端**
```python
def classify_question(state):
    llm = Tongyi(...)        # 每次调用都新建
def rag_policy_retrieval(state):
    milvus_client = MilvusClient(...)  # 每次调用都新建连接
    embeddings = DashScopeEmbeddings(...)  # 每次调用都新建
```
一次完整对话会创建 3-4 个 MilvusClient、3-4 个 DashScopeEmbeddings、2个 LLM 实例。对象创建 + 连接建立的开销累计非常高。

**7. thread_id 用 user_id，会话无法隔离**
```python
thread_id = user_id  # rag_service.py 第51行
```
用户 A 打开两个浏览器窗口聊天，两个窗口共享同一个会话记忆，相互串扰。

**8. `check_user_access_level` 权限逻辑有误（`auth_service.py` 第237-238行）**
```python
if required_access_level == "internal":
    return True  # 所有人都能访问 internal???
```
`internal` 级别的文档应该只有内部员工可以看，这里直接返回 True，任何人都能访问，权限形同虚设。

**9. import 放在函数内部**
```python
def rag_policy_retrieval(state):
    from pymilvus import MilvusClient         # 放在函数里
    from langchain_community.embeddings import DashScopeEmbeddings  # 放在函数里
```
Python 会每次调用都执行 import 语句（虽然有缓存，但有查找开销），代码风格不规范。

---

### 🔵 工程问题（不影响功能，但影响可维护性）

**10. 大量注释代码污染文件**：`rag_service.py` 有超过100行注释掉的旧代码，没有任何价值。

**11. MILVUS_URI 到处硬编码**：`document_service.py` 第16行、`rag_graph_ai.py` 第18行各自硬编码了不同的 IP。

**12. 切片器是自己实现的简陋版本**：自写切割器没有处理好 overlap 边界的语义，应使用 LangChain 的 `RecursiveCharacterTextSplitter`。

**13. PDF 解析用 PyPDF2**：这个库已经多年不维护，中文 PDF 提取效果差，应换成 `pymupdf`。

**14. Basic Authorization 硬编码**：`"Authorization":"Basic c2FiZXIzOnNhYmVyM19zZWNyZXQ="` 出现在两处代码里，是敏感信息，应放 `.env`。

## 分阶段优化计划

结合你的学习阶段，按"**先堵漏洞 → 再学新东西 → 最后工程化**"的顺序推进：

---

### 阶段 A：Bug 修复（1周，不涉及新知识）

> 这些是必须做的，任何优化建立在数据正确的基础上

| 优先 | 问题              | 改法                                   | 学到什么          |
| ---- | ----------------- | -------------------------------------- | ----------------- |
| ①    | chunk ID 碰撞     | 改成 `f"{id}_{i}"` 字符串 ID           | Milvus 主键设计   |
| ②    | user_info 作用域  | WebSocket 里加 `user_info = {}` 初始化 | Python 变量作用域 |
| ③    | role_id KeyError  | 改成 `.get("role_id", 0)`              | 防御性编程        |
| ④    | internal 权限逻辑 | 修复 `check_user_access_level`         | RBAC 权限模型     |
| ⑤    | 硬编码敏感信息    | 统一移到 `.env`                        | 环境变量管理      |

---

### 阶段 B：知识库优化（2周，核心技术点）

> 这是整个系统的数据质量基础，直接影响 RAG 回答质量

**B1: 切片器升级**
把自写切割器换成 `RecursiveCharacterTextSplitter`，理解 `chunk_size`、`chunk_overlap`、`separators` 的含义，体会不同切法对检索质量的影响。这是 RAG 优化的核心基础知识。

**B2: Milvus 显式 Schema**
为 `document_collection` 建立正式 Schema，给 `type`、`access_level`、`record_id` 字段建立标量索引。这让你真正理解 Milvus 的数据模型，而不是把它当黑盒用。

```python
# 你现在
milvus_client.create_collection(collection_name, dimension=1024)

# 应该是
schema = CollectionSchema(fields=[
    FieldSchema("id", DataType.VARCHAR, is_primary=True, max_length=64),
    FieldSchema("vector", DataType.FLOAT_VECTOR, dim=1024),
    FieldSchema("text", DataType.VARCHAR, max_length=65535),
    FieldSchema("record_id", DataType.VARCHAR, max_length=64),
    FieldSchema("type", DataType.VARCHAR, max_length=32),
    FieldSchema("access_level", DataType.VARCHAR, max_length=32),
])
```

**B3: PDF 解析器升级**
换掉 PyPDF2，改用 `pymupdf`（`import fitz`），对比两者在你公司的 PDF 文档上的提取效果，直观感受差距。

---

### 阶段 C：聊天交互优化（2-3周，LangGraph 深入）

> 这是业务价值最高的部分，也是你学习重点

**C1: 真正的流式输出**

这是最重要的用户体验优化。目前是"等全文生成 → 分块发送"，应该改成"LLM 生成 token → 立即发送"。

实现方式：在 `generate_answer_with_rag` 里让 LLM 以流式模式调用，把 answer 生成改成 generator，在 `rag_service` 里通过 `stream_mode="messages"` 捕获每个 token。这涉及理解 LangGraph 的 `astream_events`，是进阶知识。

**C2: LLM 和 Milvus 客户端单例化**

把分散在各节点内的对象创建移到模块级别，理解 Python 模块单例的原理。

**C3: 会话管理**

用 `user_id + session_id` 组合作为 `thread_id`，支持用户多会话。这要求前端传 `session_id`，是一个完整的功能迭代。

---

### 阶段 D：系统工程化（1-2周，拉平与企业的差距）

> 这部分让你的代码从"学生项目"变成"可维护的工程"

**D1: 统一配置管理**
所有 MILVUS_URI、JAVA_SERVICE_URL 等配置集中到一个 `config.py`，各模块从这里 import。

**D2: 结构化日志**
把所有 `print(f"...")` 替换成 `logger.info/debug/warning`，了解日志分级的意义。

**D3: 文档上传改异步**
`upload_document_data` 是耗时的同步函数（下载 + 解析 + embedding + 写库），放在 FastAPI 的 async 请求里会阻塞整个事件循环。改成用 `asyncio.to_thread` 或背景任务处理。

**D4: 清理注释代码**
这不是可选项，是基本代码卫生。

---

## 优先级汇总

```
立刻做 (阶段A, 1周)
  → chunk ID bug + user_info 作用域 + 权限逻辑修复

紧接着 (阶段B, 2周)
  → RecursiveCharacterTextSplitter + Milvus 显式 Schema + ID 统一

然后 (阶段C, 2-3周)
  → 真流式输出 (最难，最有价值) + 单例化 + 会话隔离

最后 (阶段D, 1-2周)
  → 配置集中管理 + 日志规范 + 异步上传 + 清理注释
```

**最需要注意的一点：** 阶段 B 的 Milvus Schema 改造，需要把现有 collection 删掉重建，意味着已经导入的文档向量数据要重新导入。这要在数据量还小的时候做，越晚做代价越大。
